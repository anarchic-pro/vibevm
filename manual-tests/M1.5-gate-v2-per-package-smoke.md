# M1.5-gate v2 — per-package registry smoke test

## Purpose

Prove the **decentralized per-package** registry shape end-to-end on
the real `vibespecs` organization at GitVerse: `flow:wal`,
`flow:sync-from-code`, `flow:atomic-commits` each live in their own
repo (`vibespecs/flow-<name>.git`), each carries its content at the
repo root, each is tagged `v0.1.0`. A consumer project resolves all
three through `[[registry]] url = "git@gitverse.ru:vibespecs"`,
installs them, lockfile v2 is fully populated, `vibe registry sync`
refreshes per-package clones, symmetric uninstall returns user-owned
files byte-identical.

This test is the per-package counterpart of the original
[M1.5-gate-multi-package-smoke.md](M1.5-gate-multi-package-smoke.md)
(which exercised the legacy single-repo `anarchic/vibespecs.git`
monorepo). The two are kept side by side until the legacy registry
is decommissioned — running both confirms M1.1-revision did not break
anyone still pinned at the v1 schema.

Run this:

- After live-publishing each of the three packages into `vibespecs/`
  via `vibe registry publish` (the live-migration commit that lands
  with this smoke).
- Before tagging M1.1-revision complete.
- After any change touching `vibe-registry::GitPackageRegistry`,
  `MultiRegistryResolver`, lockfile-v2 schema, or the
  `vibe registry sync` walk.
- After any rotation of `DEFAULT_REGISTRY_URL` (in
  `vibe_core::manifest::project`) — this test is what verifies the
  new default URL actually works against a live host.

## Preflight

1. **Tooling.**

   ```bash
   git --version            # 2.30+
   cargo --version          # workspace MSRV (1.93+)
   ```

2. **SSH auth to GitVerse.** `ssh -T git@gitverse.ru` should confirm
   authentication (and report no shell access — that's expected).
   The same SSH key that pushes to `anarchic/vibevm` is what fetches
   from `vibespecs/...`.

3. **All three per-package repos exist.** Visit (or `git ls-remote`):

   ```bash
   git ls-remote --tags git@gitverse.ru:vibespecs/flow-wal.git
   git ls-remote --tags git@gitverse.ru:vibespecs/flow-sync-from-code.git
   git ls-remote --tags git@gitverse.ru:vibespecs/flow-atomic-commits.git
   ```

   Each must list at least `v0.1.0`. If any is missing, the
   live-migration step failed; re-run the relevant `vibe registry
   publish` invocation before continuing this smoke.

4. **vibevm built release.**

   ```bash
   cargo build --release --workspace
   ```

5. **Scratch dir + cache isolation.**

   ```bash
   SCRATCH="$(mktemp -d)"
   export VIBE_REGISTRY_CACHE="$SCRATCH/registries"
   PROJECT="$SCRATCH/proj"
   VIBE="$(pwd)/target/release/vibe"
   if [ ! -x "$VIBE" ]; then VIBE="$VIBE.exe"; fi
   "$VIBE" --version
   ```

   The user's real `~/.vibe/` MUST NOT be touched.

## Steps

### 1. Scaffold + per-package registry default

```bash
"$VIBE" init --path "$PROJECT" \
    --registry-url "git@gitverse.ru:vibespecs"
grep -A4 '^\[\[registry\]\]' "$PROJECT/vibe.toml"
```

**Expected.** `vibe.toml` carries the new `[[registry]]` array form:

```toml
[[registry]]
name = "default"
url = "git@gitverse.ru:vibespecs"
```

The `naming` field is omitted because `kind-name` is the default
(skip-on-serialize). Once `DEFAULT_REGISTRY_URL` rotates to
`git@gitverse.ru:vibespecs`, the `--registry-url` override becomes
unnecessary; until then this is the explicit invocation.

### 2. Install all three packages

```bash
"$VIBE" install flow:wal             --path "$PROJECT" --assume-yes
"$VIBE" install flow:sync-from-code  --path "$PROJECT" --assume-yes
"$VIBE" install flow:atomic-commits  --path "$PROJECT" --assume-yes
```

**Expected.**

- Each install resolves through the org URL, composes
  `<org>/<kind>-<name>.git`, fetches via `git ls-remote --tags` +
  `git clone --branch v0.1.0`, materialises content into
  `<project>/.vibe/cache/<kind>/<name>/v0.1.0/`, and writes plan
  files plus the boot snippet into the project tree.
- All three exit 0.
- Boot-snippet prefixes `10-` / `20-` / `30-` install side by side
  without collision.

### 3. Inspect lockfile — schema v2

```bash
cat "$PROJECT/vibe.lock"
```

**Expected.** `[meta]` carries:

```toml
[meta]
schema_version = 2
generated_by = "vibe ..."
generated_at = "..."
root_dependencies = ["flow:wal", "flow:sync-from-code", "flow:atomic-commits"]
```

`[[package]]` entries — three of them — each carry the full v2
provenance shape. Critical fields per entry:

```toml
[[package]]
kind = "flow"
name = "wal"
version = "0.1.0"
registry = "default"
source_url = "git@gitverse.ru:vibespecs/flow-wal.git"
source_ref = "v0.1.0"
content_hash = "sha256:..."
```

`source_ref` MUST be `v0.1.0` exactly — this is the per-package
shape, not the legacy `#flow/wal/v0.1.0` fragment form. The
`registry` field MUST be `"default"` matching the `[[registry]].name`
in `vibe.toml`. `overridden` is absent (false, skip-on-serialize).

### 4. Cache layout

```bash
ls "$VIBE_REGISTRY_CACHE"
ls "$VIBE_REGISTRY_CACHE"/*/packages/
ls "$VIBE_REGISTRY_CACHE"/*/packages/flow-wal/clone/
```

**Expected.**

- Exactly **one** subdirectory under `$VIBE_REGISTRY_CACHE` — all
  three packages share the same registry-bucket because they share
  the same canonical org URL. Bucket name is the 16-hex prefix of
  `sha256(normalize("git@gitverse.ru:vibespecs"))`.
- Inside the bucket, `packages/` holds three subdirs: `flow-wal/`,
  `flow-sync-from-code/`, `flow-atomic-commits/`. Each subdir holds
  one `clone/` (real git working tree at `v0.1.0`).
- The clone has `vibe-package.toml` at its **root** (per-package
  layout). The legacy `clone/flow/wal/v0.1.0/...` path from the
  monorepo smoke MUST NOT exist.

### 5. boot/ coexistence

```bash
ls "$PROJECT/spec/boot/"
```

**Expected.** Five files: `00-core.md`, `10-flow-wal.md`,
`20-flow-sync-from-code.md`, `30-flow-atomic-commits.md`,
`90-user.md`. Same as the v1 smoke — the boot-snippet model is
unchanged by the per-package refactor.

### 6. `vibe registry sync` walks per-package clones

```bash
"$VIBE" registry sync --path "$PROJECT"
```

**Expected.** Three "refreshed" lines, one per installed package, of
shape:

```
flow:wal @ v0.1.0 via registry `default`
flow:sync-from-code @ v0.1.0 via registry `default`
flow:atomic-commits @ v0.1.0 via registry `default`

vibe registry sync: 3 refreshed, 0 skipped.
```

The summary at the end shows `3 refreshed, 0 skipped`. Sync hits
each per-package clone — verify by checking one clone's reflog or
`fsck` output if you want, but the clean exit is the smoke check.

### 7. `vibe install` is idempotent — no growth on re-install

```bash
"$VIBE" install flow:wal --path "$PROJECT" --assume-yes 2>&1 | tail -5
"$VIBE" list --path "$PROJECT" | wc -l
```

**Expected.** First command exits non-zero (already-installed guard
from M0). Second command's package count remains three — the
already-installed guard fires before any file is touched.

### 8. Symmetric uninstall — user-owned bytes preserved

```bash
cp "$PROJECT/spec/boot/00-core.md" "$SCRATCH/00-core.before"
cp "$PROJECT/spec/boot/90-user.md" "$SCRATCH/90-user.before"
cp "$PROJECT/spec/WAL.md"          "$SCRATCH/WAL.before"

"$VIBE" uninstall flow:atomic-commits --path "$PROJECT" --assume-yes
"$VIBE" uninstall flow:sync-from-code --path "$PROJECT" --assume-yes
"$VIBE" uninstall flow:wal             --path "$PROJECT" --assume-yes

cmp "$PROJECT/spec/boot/00-core.md" "$SCRATCH/00-core.before" && echo "00-core intact"
cmp "$PROJECT/spec/boot/90-user.md" "$SCRATCH/90-user.before" && echo "90-user intact"
cmp "$PROJECT/spec/WAL.md"          "$SCRATCH/WAL.before"      && echo "WAL intact"
```

**Expected.** All three `cmp` lines print "intact". Lockfile's
`[[package]]` array is empty after the third uninstall. Spec/flows
subdirectories empty (or absent — both acceptable per M0 rules).

`[meta].root_dependencies` is reset to `[]` after the last uninstall
— uninstalling a root drops it from the list. (If it doesn't, file
the regression: the lockfile-v2 root-tracking is busted.)

### 9. v1 lockfile back-compat probe (optional)

If you have access to a v1-shaped lockfile (any project from before
M1.1-revision shipped), drop a copy at `$PROJECT/vibe.lock` and re-run
`vibe install flow:wal --path "$PROJECT" --assume-yes`. The lockfile
on disk after this command must have `schema_version = 2` and the
fields rewritten in the modern shape — v1 → v2 migration is on the
read path.

Skip if no v1 lockfile is around.

### 10. `cargo test` once more

```bash
cargo test --workspace --quiet
```

**Expected.** All green. If anything in this smoke left global state
in a weird shape, this catches it.

## Cleanup

```bash
rm -rf "$SCRATCH"
unset VIBE VIBE_REGISTRY_CACHE SCRATCH PROJECT
ls ~/.vibe/registries 2>/dev/null || echo "no ~/.vibe/registries — good"
```

## What to file if it fails

1. `vibe --version`, `git --version`, `cargo --version`.
2. Failing step's full console output (stdout + stderr).
3. `cat "$PROJECT/vibe.toml"` and `cat "$PROJECT/vibe.lock"`.
4. `ls -R "$VIBE_REGISTRY_CACHE"` (full tree) — particularly important
   if the per-package `packages/<kind>-<name>/clone/` shape doesn't
   appear.
5. For step 6 (`registry sync`): if any package was reported skipped,
   the full `RefreshedVia` reason string per skipped entry.
6. For step 8 (uninstall mismatch): the `cmp` exit code and the
   diff-able offending file.
7. The output of `git ls-remote --tags <repo>` for any package whose
   tag/commit assertions failed in step 3 — distinguishes "registry
   moved tag" (force-push by maintainer) from "client misread tag"
   (vibevm bug).

## Notes on running this against the legacy registry

If you point this smoke at `git@gitverse.ru:anarchic/vibespecs.git`
(the v1 monorepo URL) instead of `git@gitverse.ru:vibespecs` (the v2
org URL), the resolver will compose
`anarchic/vibespecs/flow-wal.git` — which doesn't exist, since the v1
URL is itself a single repo. You will see `RegistryError::UnknownPackage`
for each install. That's expected behaviour; the v1 monorepo
URL is no longer a valid `[[registry]].url` for the per-package
model. Use the legacy [M1.5-gate-multi-package-smoke.md](M1.5-gate-multi-package-smoke.md)
to exercise that path.
