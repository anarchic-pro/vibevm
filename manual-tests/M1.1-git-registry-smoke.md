# M1.1 — git-registry smoke test

## Purpose

Prove the M1.1 git-backed registry path end-to-end against the real
public registry at `git@gitverse.ru:anarchic/vibespecs.git`. The
automated `cargo test` suite covers the same flow against a
`git+file://` bare repo — this test covers what that cannot: real
SSH authentication, real network, real-world `~/.vibe/` layout, and
the exact lockfile `source_uri` we hand to consumers.

Run this before tagging M1.1. Re-run whenever the registry backend,
the CLI arg parsing, or the lockfile schema changes.

**Last known pass.** 2026-04-22 against `vibespecs@98e51fc` on
Windows 11 / Git Bash, with `git` 2.x and Rust 1.93. Record the next
successful run here — date, commit of vibespecs, OS, git version —
so a future session can see this wasn't a one-off.

## Preflight

Verify each of these on the machine that will run the test, in
order. If any fails, stop — this test cannot proceed.

1. **Tooling.**
   ```bash
   git --version            # any reasonably recent release; 2.30+ is fine
   cargo --version          # matches the workspace MSRV (1.93+ today)
   ```

2. **Network + SSH auth to GitVerse.** Expect the
   "successfully authenticated, but Gitverse does not provide shell
   access" message — that means auth worked and the server refused
   the shell, which is what we want.
   ```bash
   ssh -T git@gitverse.ru
   ```
   If you see `Permission denied (publickey)`, fix your SSH setup
   before continuing — vibe's shell-out backend will inherit the same
   identity and hit the same wall.

3. **Working tree of vibevm itself.** You must be inside a clean
   checkout of the vibevm source repo with the workspace built.
   ```bash
   cd /path/to/vibevm            # the repo that contains crates/, spec/, etc.
   git status                    # should be clean or only expected WIP
   cargo build --release -p vibe-cli
   ```
   The subsequent steps all shell out to
   `./target/release/vibe` (or `./target/release/vibe.exe` on
   Windows).

## Setup — clean slate

These exports isolate the entire run from your real `~/.vibe/`
directory and from any prior scratch state. Copy-paste the whole
block.

```bash
SCRATCH="$(mktemp -d -t vibevm-smoke.XXXXXX)"
export VIBE_REGISTRY_CACHE="$SCRATCH/registries"
PROJECT="$SCRATCH/proj"
echo "SCRATCH   = $SCRATCH"
echo "PROJECT   = $PROJECT"
echo "REG_CACHE = $VIBE_REGISTRY_CACHE"
```

**Why this is the clean slate.** `VIBE_REGISTRY_CACHE` redirects the
per-user registry cache away from `~/.vibe/registries/` into
`$SCRATCH`. Nothing under `~/` is touched for the duration of the
test. `$PROJECT` is a fresh empty directory — no pre-existing
lockfile, no pre-existing `vibe.toml`.

### Pick the vibe binary

```bash
VIBE="$(pwd)/target/release/vibe"
# On Windows (Git Bash): the file is actually vibe.exe; both spellings work.
if [ ! -x "$VIBE" ]; then VIBE="$VIBE.exe"; fi
"$VIBE" --version            # prints `vibe 0.1.0-dev` or similar
```

## Steps

### 1. Scaffold a fresh project

```bash
"$VIBE" init --path "$PROJECT"
```

**Expected.** One line per created file (`spec/boot/00-core.md`,
`spec/boot/90-user.md`, `vibe.toml`, `.gitignore`, etc.), a trailing
"Initialized vibevm project" summary, exit code 0.

### 2. Confirm `vibe init` pre-populated `[registry]`

Since 2026-04-22, `vibe init` writes a default `[registry]` section
pointing at the public GitVerse registry unless `--no-registry` or
`--registry-url <URL>` is passed. For this smoke-test the default is
exactly what we want — no manual edit needed.

```bash
cat "$PROJECT/vibe.toml"
```

**Expected.** The manifest contains:

```toml
[registry]
url = "git@gitverse.ru:anarchic/vibespecs.git"
```

The `ref` line is absent — `main` is the default and serde skips
it on serialize. If you need a different ref, re-run step 1 with
`--registry-ref <REF>`.

If you are running this test against a private / staging registry,
skip step 1's `vibe init` above and instead use:

```bash
"$VIBE" init --path "$PROJECT" --registry-url "<your-url>" --registry-ref "<your-ref>"
```

…then come back here.

### 3. Install flow:wal from the real registry

```bash
"$VIBE" install flow:wal --path "$PROJECT" --assume-yes
```

**Expected.**

- The first-line heading says `Resolving flow:wal…`.
- Tracing log (visible if you set `VIBE_LOG=info`) mentions
  "cloning registry" with `url=git@gitverse.ru:anarchic/vibespecs`
  pointing at a path under `$VIBE_REGISTRY_CACHE`.
- A `Plan for flow:wal@0.1.0` block listing the writes
  (`spec/flows/wal/WAL-PROTOCOL.md`, `spec/flows/wal/*`,
  `spec/boot/10-flow-wal.md`).
- `Installed 1 package (N files written).` summary.
- Exit code 0.

If auth fails the error message will be
`ssh authentication failed for git@gitverse.ru:...`. Fix your SSH
setup and rerun from step 1.

### 4. Inspect the lockfile

```bash
cat "$PROJECT/vibe.lock"
```

**Expected.** One `[[package]]` entry with

```toml
kind = "flow"
name = "wal"
version = "0.1.0"
source = "git+ssh://git@gitverse.ru/anarchic/vibespecs.git#flow/wal/v0.1.0"
content_hash = "sha256:..."
boot_snippet = "10-flow-wal.md"
```

The `source` value is the critical one — the `git+ssh://` prefix
proves `resolve_registry` routed through `GitRegistry`, and the
`#flow/wal/v0.1.0` fragment proves `GitRegistry::fetch` stamped the
right target.

### 5. Inspect the registry cache layout

```bash
ls "$VIBE_REGISTRY_CACHE"
ls "$VIBE_REGISTRY_CACHE"/*/
cat "$VIBE_REGISTRY_CACHE"/*/meta.toml
```

**Expected.**

- Exactly one subdirectory under `$VIBE_REGISTRY_CACHE`, named with
  16 hex characters.
- That directory contains `clone/` (a real git working tree — `ls
  clone/.git` should show git's internals) and `meta.toml`.
- `meta.toml` carries a normalized `url` (lowercase,
  no trailing `.git`), `ref = "main"`, a recent `last_pulled_at`
  (UTC RFC-3339), and `url_hash` (full 64-char sha256 hex).

### 6. `vibe list` reflects the install

```bash
"$VIBE" list --path "$PROJECT"
```

**Expected.** Table with one row: kind `flow`, name `wal`, version
`0.1.0`, boot snippet `10-flow-wal.md`.

### 7. Freshness: re-install does not re-pull

The TTL defaults to 1 hour; an install that runs within that window
must not touch the network.

```bash
META="$(ls -d $VIBE_REGISTRY_CACHE/*/meta.toml)"
before=$(stat -c %Y "$META" 2>/dev/null || stat -f %m "$META")
"$VIBE" uninstall flow:wal --path "$PROJECT" --assume-yes
"$VIBE" install   flow:wal --path "$PROJECT" --assume-yes
after=$(stat -c %Y "$META" 2>/dev/null || stat -f %m "$META")
echo "meta.toml mtime before=$before after=$after"
```

**Expected.** `before` equals `after` — the second install reused the
cached clone and did not touch `meta.toml`. (The GNU `stat -c %Y`
form works in Git Bash on Windows and on Linux; the `stat -f %m`
fallback is for macOS.)

### 8. `vibe registry sync` always pulls

```bash
sleep 2                       # so mtime resolution catches the update
"$VIBE" registry sync --path "$PROJECT"
synced=$(stat -c %Y "$META" 2>/dev/null || stat -f %m "$META")
echo "meta.toml mtime after sync=$synced (was $after)"
```

**Expected.** `synced > after` — `sync` forced a fetch + reset and
rewrote `meta.toml.last_pulled_at`. The console summary ends with
`Registry cache up to date at …`.

### 9. Uninstall

Before removing, snapshot the user-owned files so we can prove
`uninstall` did not touch them.

```bash
cp "$PROJECT/spec/boot/00-core.md"  "$SCRATCH/00-core.before"
cp "$PROJECT/spec/boot/90-user.md"  "$SCRATCH/90-user.before"
"$VIBE" uninstall flow:wal --path "$PROJECT" --assume-yes
cmp "$PROJECT/spec/boot/00-core.md"  "$SCRATCH/00-core.before"  && echo "00-core intact"
cmp "$PROJECT/spec/boot/90-user.md"  "$SCRATCH/90-user.before"  && echo "90-user intact"
```

**Expected.**

- Every file listed in step 3's plan is removed.
- Both `cmp` calls print their "intact" line — user-owned boot files
  are byte-for-byte identical to their pre-uninstall state.
- `vibe.lock` still exists but its `[[package]]` array is empty.

```bash
"$VIBE" list --path "$PROJECT"    # should say "(no packages installed)"
```

### 10. `cargo test` once more, for belt-and-braces

```bash
cargo test --workspace --quiet
```

**Expected.** All green. If a manual-test step above left the
workspace in a strange state (e.g. a stray lockfile somewhere under
`tests/`), this catches it.

## Cleanup

```bash
rm -rf "$SCRATCH"
unset VIBE VIBE_REGISTRY_CACHE SCRATCH PROJECT META
```

Confirm:

```bash
ls ~/.vibe/registries 2>/dev/null || echo "no ~/.vibe/registries — good"
```

The test is designed so this directory either did not exist before
the run and still does not, or existed and is byte-identical to its
pre-run state. If you see a new entry there, some step spawned a
`vibe` invocation without `VIBE_REGISTRY_CACHE` in the environment —
investigate before closing the test.

## What to file if it fails

Copy the following into the bug report / issue:

1. `vibe --version` output.
2. `git --version` output and `ssh -T git@gitverse.ru` output.
3. The failing step's full console output, including stderr.
4. `cat "$PROJECT/vibe.toml"` and `cat "$PROJECT/vibe.lock"`.
5. `ls -R "$VIBE_REGISTRY_CACHE"` and
   `cat "$VIBE_REGISTRY_CACHE"/*/meta.toml`.
6. If the failure looks registry-side, rerun the failing step with
   `VIBE_LOG=debug` in the environment and attach the stderr.

Do not delete `$SCRATCH` until the bug is filed — the state in it is
the diagnostic.
