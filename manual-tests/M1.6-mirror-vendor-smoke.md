# M1.6 — Multi-registry, mirror, and vendor smoke test

## Purpose

Prove Phase B v0 of [PROP-002](../spec/modules/vibe-registry/PROP-002-decentralized-registry.md) end-to-end on a real machine. Three distinct contracts get exercised together because they form the cohesive "mirrors are now useful for actual installs" story:

- **Multi-`[[registry]]` priority walk** (PROP-002 §2.2). Two registries; the first is empty, the second has the package; `MultiRegistryResolver` walks them in order and the second wins on `UnknownPackage` fall-through.
- **`[[mirror]]` dispatch on the cache-mutating fetch path** (PROP-002 §2.3). When the canonical primary URL becomes unreachable, the registered mirror takes over; the lockfile still records the canonical URL as `source_url`.
- **`vibe registry vendor` → wire as `file://` `[[mirror]]`** (PROP-002 §6, Phase B preview). Vendor a freshly installed project; wire the output dir as a `file://` mirror; reach offline / air-gapped fetch by deliberately breaking the network primary.

This complements [M1.5-gate-v2-per-package-smoke.md](M1.5-gate-v2-per-package-smoke.md) which exercises one live registry without mirrors. Run this **after** M1.5-gate-v2 passes — Phase B v0 builds on the per-package model M1.5-gate-v2 covers.

Run this:

- After tagging M1.6 ready.
- After any change touching `MultiRegistryResolver`, the `GitPackageRegistry::fetch` / `refresh_package` mirror walk, `bootstrap_or_update_at`, the `fetch_with_expected_hash` cross-source gate, the `vibe registry vendor` pipeline, or `[[mirror]]` parsing.
- After any rotation of `DEFAULT_REGISTRY_URL` — Scenario A's primary URL must stay reachable.

### Known regressions surfaced by this smoke

- **B4 fall-through discriminator over-strict on connect-failure** (2026-05-04, vibevm `82f32cc`). When the primary registry's URL points at a dead host (DNS resolves to nothing reachable / connect refused), `git ls-remote --tags` exits with `fatal: unable to access '…': Failed to connect to … port 443 after Nms: Could not connect to server`. The classifier in `crates/vibe-registry/src/git_backend/shell.rs::classify_clone_error` matches `could not resolve host` / `could not read from remote repository` / `network is unreachable` but **not** `failed to connect` / `could not connect to server`, so the failure lands as `GitError::CommandFailed` rather than `NetworkUnreachable` or `RepoNotFound`. Even if the classifier were widened, `MultiRegistryResolver` only translates `RepoNotFound` into `UnknownPackage` for the priority walk (`git_package_registry.rs:411`), so a connect-failure on the primary halts install rather than falling through to the fallback registry. Two-part decision is required: (a) widen the classifier substring set, (b) decide whether connect-failure on the primary should be transient-fall-through or hard-fail. Tracked in WAL "Known issues".

## Scenario A — vendor → file:// mirror → offline install

### A1. Install all three flows from the GitHub primary

```bash
"$VIBE" init --path "$PROJECT"
"$VIBE" install flow:wal flow:sync-from-code flow:atomic-commits --path "$PROJECT" --assume-yes
```

**Expected.** Three packages install. `vibe.lock` carries `schema_version = 2`, three `[[package]]` entries, each with `registry = "vibespecs"` and `source_url = "https://github.com/vibespecs/flow-<name>.git"`. (See M1.5-gate-v2 steps 2–3 for the full lockfile shape.)

### A2. Vendor everything in the lockfile

```bash
"$VIBE" registry vendor --out "$VENDOR" --path "$PROJECT"
ls "$VENDOR"
```

**Expected.** The vendor dir contains a bare repo per package plus a generated README:

```
flow-atomic-commits.git/
flow-sync-from-code.git/
flow-wal.git/
README.md
```

The vendor command prints a suggested `[[mirror]]` URL of the form `file:///<abs>/vendor-mirror`.

### A3. Verify each vendored repo is a usable git source

```bash
for pkg in flow-wal flow-sync-from-code flow-atomic-commits; do
  echo "=== $pkg ==="
  git ls-remote --tags "$VENDOR/$pkg.git"
done
```

**Expected.** Each `ls-remote` reports at least `refs/tags/v0.1.0`. Peeled SHAs (the `^{}` rows) match what `git ls-remote --tags https://github.com/vibespecs/flow-<name>` reports — proves the vendor preserved identity, not just shape.

### A4. Wire the vendor dir as a `[[mirror]]`

```bash
# Build the file:// URL portably. On Git Bash / MSYS the ABS path
# starts with /c/Users/...; convert to C:/Users/... so the URL is
# valid on Windows too.
ABS_VENDOR="$(cd "$VENDOR" && pwd)"
case "$ABS_VENDOR" in
  /[a-z]/*) ABS_VENDOR="$(echo "$ABS_VENDOR" | sed 's@^/\([a-z]\)/@\1:/@')" ;;
esac

"$VIBE" registry set-mirror vibespecs "file:///$ABS_VENDOR" --priority 0 --path "$PROJECT"
"$VIBE" registry list --path "$PROJECT"
```

**Expected.** `registry list` output shows the `vibespecs` primary plus the `file://` mirror at priority 0 attached to it.

### A5. Make the primary unreachable, verify mirror takes over

Replace the primary URL with a deliberately invalid host. The mirror chain remains intact.

```bash
cp "$PROJECT/vibe.toml" "$PROJECT/vibe.toml.original"
sed -e 's|https://github.com/vibespecs|https://invalid.example/vibespecs|' \
    "$PROJECT/vibe.toml" > "$PROJECT/vibe.toml.broken"
```

Spin up a fresh project carrying the broken-primary + working-mirror manifest. The fresh project has an empty lockfile, so install re-fetches every package.

```bash
mkdir -p "$RESCUE"
cp "$PROJECT/vibe.toml.broken" "$RESCUE/vibe.toml"
mkdir -p "$RESCUE/spec/boot"
: > "$RESCUE/spec/boot/00-core.md"
: > "$RESCUE/spec/boot/90-user.md"
: > "$RESCUE/spec/WAL.md"

# Wipe the registries cache so install really re-fetches; otherwise
# the existing clone from step A1 would mask whether the mirror walk
# actually fired.
rm -rf "$VIBE_REGISTRY_CACHE"

RUST_LOG=vibe_registry=info "$VIBE" install flow:wal --path "$RESCUE" --assume-yes 2>&1 | tee "$SCRATCH/rescue.log"
```

**Expected.**

- Install **succeeds** despite the broken primary URL — the file:// mirror serves.
- The log captures at least one `fetch served by mirror` event from `vibe_registry`.
- `RESCUE/vibe.lock` `[[package]]` entry for `flow:wal` carries:
  - `registry = "vibespecs"` (canonical registry name).
  - `source_url = "https://invalid.example/vibespecs/flow-wal.git"` — the **canonical primary URL**, NOT the mirror URL. PROP-002 §2.3 step 3.
  - `content_hash` byte-identical to the one A1 produced for `flow:wal` in `$PROJECT/vibe.lock` — proves cross-source identity.

```bash
diff <(grep '^content_hash' "$PROJECT/vibe.lock") \
     <(grep '^content_hash' "$RESCUE/vibe.lock")
# expected: empty diff (same content_hash for the same package)
```

### A6. Restore the primary, confirm symmetric reverse

Sanity: with the canonical primary back, install still works (mirror is fallback only).

```bash
cp "$PROJECT/vibe.toml.original" "$PROJECT/vibe.toml"
rm -rf "$VIBE_REGISTRY_CACHE" "$RESCUE"
```

(Scenario A done — Scenario B re-uses `$VIBE_REGISTRY_CACHE` which is now empty.)

## Scenario B — multi-`[[registry]]` priority walk

### B1. Build two local per-package registries

`reg-a` is empty (no packages); `reg-b` carries `flow-wal.git` built from the in-tree fixture.

```bash
REG_A="$SCRATCH/reg-a"
REG_B="$SCRATCH/reg-b"
SRC="$SCRATCH/src-wal"
mkdir -p "$REG_A" "$REG_B"

cp -R "$REPO/fixtures/registry/flow/wal/v0.1.0" "$SRC"
( cd "$SRC" \
  && git init -q --initial-branch=main \
  && git config user.email t@example.com \
  && git config user.name Test \
  && git add -A \
  && git commit -q -m "0.1.0" \
  && git tag v0.1.0 )
git clone --bare -q "$SRC" "$REG_B/flow-wal.git"

ABS_A="$(cd "$REG_A" && pwd)"; ABS_B="$(cd "$REG_B" && pwd)"
case "$ABS_A" in /[a-z]/*) ABS_A="$(echo "$ABS_A" | sed 's@^/\([a-z]\)/@\1:/@')" ;; esac
case "$ABS_B" in /[a-z]/*) ABS_B="$(echo "$ABS_B" | sed 's@^/\([a-z]\)/@\1:/@')" ;; esac
```

### B2. Scaffold a project with both registries — empty one first

```bash
"$VIBE" init --path "$PROJECT_B" --no-registry
cat >> "$PROJECT_B/vibe.toml" <<EOF

[[registry]]
name = "primary-empty"
url = "file:///$ABS_A"

[[registry]]
name = "fallback"
url = "file:///$ABS_B"
EOF

"$VIBE" registry list --path "$PROJECT_B"
```

**Expected.** Two registries listed, primary first, both with `naming = "kind-name"`.

### B3. Install — primary misses, fallback serves

```bash
RUST_LOG=vibe_registry=info "$VIBE" install flow:wal --path "$PROJECT_B" --assume-yes 2>&1 | tee "$SCRATCH/proj-b.log"
"$VIBE" list --path "$PROJECT_B"
```

**Expected.**

- `flow:wal` installs successfully.
- `vibe.lock` entry for `flow:wal`:
  - `registry = "fallback"` — recorded as the registry that actually served.
  - `source_url = "file:///<ABS_B>/flow-wal.git"` — the URL composed from the **fallback** registry's org URL.
- The resolver's primary→fallback walk shows up in the resolved-graph output (the `Resolving N root packages…` step lands the package against `fallback`, not `primary-empty`).

### B4. (Optional) Confirm primary failure mode is `UnknownPackage`-only

If the primary fails for any reason **other than** `UnknownPackage` (network, auth, malformed manifest), the resolver MUST surface the error rather than silently falling through to `fallback`. To verify the discriminator works, point `primary-empty` at a non-existent host:

```bash
sed -i.bak "s@file:///$ABS_A@https://invalid.example/empty@" "$PROJECT_B/vibe.toml"
"$VIBE" install flow:wal --path "$PROJECT_B" --assume-yes 2>&1 | tail -10
```

**Expected.** The resolver still falls through — a hostile DNS / 4xx counts as `RepoNotFound`, which `MultiRegistryResolver` translates to `UnknownPackage` for the priority-walk discriminator. If you see a hard install failure here instead of a successful fall-through to the fallback, the per-package classifier is over-strict; file the regression.

```bash
mv "$PROJECT_B/vibe.toml.bak" "$PROJECT_B/vibe.toml"
```

## What to file if it fails

1. `vibe --version`, `git --version`, `cargo --version`, `uname -srm`.
2. Failing step's full console output (stdout + stderr).
3. `cat "$PROJECT/vibe.toml"` and `cat "$PROJECT/vibe.lock"` for the relevant project.
4. `ls -R "$VENDOR"` for vendor failures (Scenario A2 / A3).
5. For Scenario A5: the contents of `"$SCRATCH/rescue.log"` (the `RUST_LOG=vibe_registry=info` capture). The mirror walk's `tracing::info!` / `tracing::debug!` lines tell you whether dispatch reached the mirror at all.
6. For Scenario B3: the same log, plus `cat "$PROJECT_B/vibe.lock"` to see which registry the lockfile attributes the install to. A `registry = "primary-empty"` attribution against the empty registry would mean the priority-walk discriminator broke.
7. Pre/post `content_hash` mismatch (Scenario A5 diff non-empty) — paste the offending lockfile entries from both projects.
